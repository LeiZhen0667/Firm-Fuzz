# 系统流程的整体思路

## 1. 总体定位

本系统的目标是构建一个 **GPL-assisted black-box Web 服务模糊测试系统**。测试阶段不依赖固件镜像、不进行插桩、不读取目标设备运行时内存，也不假设能够调试设备；系统仅通过网络接口与真实设备交互。

系统的核心思想是：在测试前离线分析厂商公开 GPL 包中的源码、Web 页面、配置文件、CGI 源码或二进制、用户态服务程序等材料，从中提取 Web 服务相关知识，并将这些知识转化为模糊测试中的种子、模板、参数约束、变异策略、调度权重和漏洞 oracle。

因此，GPL 在本系统中不是被视为目标固件的完全等价替代品，而是一个 **概率性外部知识源**。它用于提高黑盒模糊测试的输入质量和探索效率，但运行时仍以真实设备反馈为准。

## 2. 系统主线

系统整体流程可以概括为：

```text
厂商 GPL 包
  ↓
GPL 解包与归一化
  ↓
Web 服务相关文件识别
  ↓
接口、参数、约束、sink 信息提取
  ↓
构建接口—参数—约束—sink 知识图谱
  ↓
生成语义有效的 HTTP 种子与请求模板
  ↓
基于参数类型、约束和 sink 进行定向变异
  ↓
向真实设备发送黑盒测试请求
  ↓
分析响应、状态变化、OOB 反馈和异常现象
  ↓
根据反馈更新模板分数和变异方向
  ↓
持续调度、复现、去重并输出漏洞报告
```

这条主线需要始终保持两个边界：第一，GPL 分析只发生在离线阶段，用于引导测试；第二，真实测试阶段只通过网络接口观察设备行为，不引入灰盒运行时能力。

## 3. GPL 信息进入系统的方式

GPL 包首先被解包、归一化并建立文件索引。系统需要识别其中与 Web 服务相关的材料，包括前端页面、JavaScript、CGI 目录、Web server 配置、CGI 源码、已编译 CGI 二进制、用户态服务代码、配置读写工具和系统命令调用逻辑。

这些材料会被统一整理成结构化信息，例如项目元数据、文件索引、URL 路由、handler 映射、参数集合、参数约束、危险 sink、源码与二进制字符串映射关系等。

系统后续所有测试决策都围绕这些结构化信息展开，而不是直接对 GPL 文件做零散规则匹配。

## 4. Web 服务知识抽取思路

系统需要从三个层面抽取 Web 服务信息。

第一层是前端层。系统从 HTML 表单、JavaScript 请求、AJAX、fetch、XMLHttpRequest、jQuery 调用中提取 URL、HTTP 方法、参数名、页面上下文和可能的认证需求。

第二层是 Web server 配置层。系统解析 CGI alias、handler 映射、URL rewrite、静态目录、认证规则等信息，用于建立外部 URL 与后端处理程序之间的关系。

第三层是后端 CGI 或服务端代码层。系统从源码或二进制字符串中识别 handler 注册、参数读取函数、条件判断、配置读写、命令执行、文件操作和危险内存操作。该层是参数约束推断和漏洞风险评估的主要来源。

最终目标不是简单获得 URL 列表，而是形成如下关系：

```text
URL → handler → 参数 → 类型/约束 → 数据流 → sink → 风险标签
```

这条关系链是后续种子构建、变异和调度的核心依据。

## 5. 参数和约束的建模思路

系统需要把从 GPL 中发现的参数转化为可执行的模糊测试知识。

参数名来自前端表单、JS 请求、CGI 参数读取函数、配置读写函数和二进制字符串。参数类型由代码行为推断，例如 `atoi` 表示整数，`inet_addr` 表示 IPv4，`strcmp` 分支表示枚举，`strlen` 检查表示长度约束，`sscanf` 表示结构化格式。

参数约束包括合法取值、边界长度、结构格式、编码方式、依赖关系和认证条件。对于结构化参数，系统应尽量保持其外层格式，只在内部字段上变异，而不是完全随机破坏结构。

参数风险由数据流和上下文决定。例如参数流向 `system/popen` 时标记为命令注入敏感，流向 `strcpy/sprintf/strcat` 时标记为溢出敏感，流向 `fopen/unlink` 时标记为路径敏感，流向 `nvram_set/uci_set` 时标记为状态改变敏感。

## 6. 种子构建的整体思路

种子不是普通的原始 HTTP 请求集合，而是带有语义信息的 HTTP 请求模板。每个模板至少包含 URL、方法、header、body/query 参数、认证需求、参数类型、合法值、边界值、风险标签、关联 sink 和来源证据。

系统生成种子时应优先满足三个目标：请求能够被目标 Web 服务接受，参数组合尽量贴近真实业务语义，高风险参数能够被明确标注并在后续变异中优先处理。

种子来源包括 GPL 前端页面、JS 请求、Web server 配置、CGI 源码、CGI 二进制字符串、真实设备爬虫抓包，以及必要时由 LLM 对非标准框架或复杂字符串拼接逻辑进行补全。

种子的最终形式应支持 Codex 后续实现为统一数据结构，例如：

```text
Template = HTTP 基本结构 + 参数元数据 + 约束信息 + sink 风险信息 + 运行反馈历史
```

## 7. 变异策略的整体思路

变异策略应由 GPL 信息驱动，而不是对参数值做无差别随机扰动。

系统首先根据参数类型执行类型感知变异。例如整数参数使用边界值、负数、极大值；字符串参数使用长度扩展、特殊字符、格式串；IPv4、MAC、路径、JSON、XML、base64、urlencode 等类型使用各自的结构保持变异。

系统其次根据 sink 执行定向变异。命令执行 sink 对应命令分隔符、OOB 回连和延时 payload；内存危险 sink 对应长度递增、格式串和边界探测；文件操作 sink 对应路径穿越、敏感文件和特殊设备路径。

系统还需要尽量保留参数格式约束。例如对于 `INT:INT:INT:STRING` 这类结构，不应直接生成随机字符串，而应保持分隔符和字段数量，只对具体字段进行边界或攻击性变异。

## 8. 黑盒执行与反馈闭环

测试阶段系统只向真实设备发送 HTTP 请求，并观察外部可见反馈。

反馈来源包括 HTTP 状态码、响应长度、响应关键词、页面跳转、错误信息、连接状态、服务可用性、设备重启、配置是否改变、认证状态是否异常，以及 DNS/HTTP/ICMP 等 OOB 回连。

系统还应利用 GPL 中提取的字符串构造伪覆盖信号。如果响应中出现 GPL 源码或二进制中的错误字符串、日志字符串或业务提示字符串，系统可以反推当前请求可能触达的函数、分支或 sink 附近逻辑。

该反馈不是严格代码覆盖率，而是黑盒条件下的近似执行线索。它的作用是帮助系统判断哪些模板更有探索价值，哪些变异方向更接近高风险路径。

## 9. 调度思路

调度器负责决定下一步测试哪个模板、哪个参数、哪类变异。

模板分数可以综合以下因素：响应是否新颖，是否命中新 GPL 字符串，是否更接近危险 sink，参数是否具有丰富约束，是否出现过异常信号，是否具有 OOB、崩溃、认证绕过或状态改变迹象。

整体策略可以分阶段进行。初始阶段偏向接口覆盖，尽量验证 GPL 中发现的 URL 是否真实存在；中期偏向 sink proximity，优先测试高风险 handler 和高风险参数；后期偏向异常信号深挖，对已经出现崩溃、延时、OOB 或状态异常的模板进行集中变异和复现。

调度算法可以先实现简单加权分数或 ε-greedy，后续再扩展为 UCB 或 Thompson Sampling。

## 10. 漏洞 oracle 的整体思路

系统的 oracle 需要适配黑盒场景。

崩溃类漏洞通过连接断开、Web 服务无响应、端口关闭、设备重启、ping 不通、HTTP 500 或稳定 reset 来判断。

命令注入优先通过 OOB DNS/HTTP 回连确认，其次通过时间延迟、响应命令输出、配置副作用和服务行为变化确认。

缓冲区溢出通过服务崩溃、设备重启、连接 reset、长度二分和稳定复现确认。

认证绕过通过未登录访问敏感接口、低权限 cookie 调用管理接口成功、缺失 token 仍然生效等现象确认。

路径穿越和文件操作类漏洞通过敏感文件内容、配置文件下载、非预期文件写入和目录枚举结果确认。

所有 oracle 都应服务于最终的复现和去重，避免只记录一次性异常。

## 11. 系统实现的总架构思路

Codex 后续实现时，可以按数据流拆分为以下主目录：

```text
gplfuzz/
  collector/      # GPL 收集、解包、归一化
  analyzer/       # Web/JS/C/二进制分析，接口、参数、约束和 sink 提取
  seed/           # HTTP 模板构建、语义补全、种子语料管理
  fuzzer/         # 变异器、调度器、会话管理、请求发送
  feedback/       # 响应差异、GPL 字符串伪覆盖、状态检查、OOB 监控
  oracle/         # 崩溃、命令注入、溢出、认证绕过、路径穿越判断
  report/         # 去重、复现脚本、漏洞报告生成
```

各模块之间应通过统一中间表示交换数据，避免模块之间直接依赖具体 GPL 文件格式。最重要的中间表示包括文件索引、接口对象、参数对象、约束对象、sink 对象、HTTP 模板对象、反馈对象和漏洞候选对象。

## 12. MVP 路线

最小可行系统不需要一开始实现完整 taint、复杂调度和完整伪覆盖。第一版可以只完成四件事。

首先，从 GPL 中提取 URL、参数和危险 sink。其次，把这些信息转换为 HTTP seed template。再次，实现基于 sink 的定向变异。最后，完成黑盒请求发送、响应记录和基础 oracle。

MVP 优先支持四类漏洞：命令注入、缓冲区溢出、路径穿越和认证绕过。

MVP 的静态分析规则可以保持轻量。例如参数附近出现 `system/popen` 就标记为命令敏感，参数进入 `strcpy/sprintf/strcat` 就标记为溢出敏感，参数进入 `fopen/unlink` 就标记为路径敏感。后续再逐步扩展跨函数数据流和更精确的约束推断。

## 13. 关键设计原则

第一，GPL 只提供引导，不提供绝对真值。任何从 GPL 得到的接口、参数和 sink 都需要在真实设备上通过探测和反馈动态校正。

第二，系统要优先生成语义有效请求。相比随机 payload，能够通过前置校验、到达后端 handler、触达危险 sink 的请求更有价值。

第三，变异应尽量围绕参数类型、格式约束和 sink 风险展开。系统的优势不是请求数量更多，而是每个请求更接近真实漏洞触发条件。

第四，反馈必须反向影响调度。响应新颖性、GPL 字符串命中、状态变化、OOB 信号和崩溃现象都应被用于提高或降低模板优先级。

第五，系统应能处理 GPL 与真实设备不一致的问题。可以通过版本匹配评分、运行时接口探测、错误知识降权、参数裁剪、同系列泛化和 UI 抓包补充来缓解。

## 14. 给 Codex 的总体实现目标

Codex 在后续开发中应围绕一个核心目标展开：

```text
把 GPL 中的 Web 服务相关源码知识，转化为可执行、可变异、可反馈优化的 HTTP fuzzing 模板。
```

所有模块都应服务于这条路径。collector 负责拿到材料，analyzer 负责提取知识，seed 负责构造模板，fuzzer 负责执行和变异，feedback 负责解释真实设备反应，oracle 负责确认漏洞信号，report 负责复现和输出证据。

最终系统不是一个单纯的 GPL 静态分析器，也不是一个普通 HTTP fuzzer，而是一个以 GPL 外部知识为引导、以真实设备黑盒反馈为闭环的 Web 服务漏洞发现系统。
